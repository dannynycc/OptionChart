# Covered Call 策略開發計畫

## GOAL

建立一個量化的 covered call 決策框架，基於主動交易者的集體部位數據（net_position），回答：
1. **賣哪裡**：哪個合約、哪個履約價（牆夠厚 × premium 值得收）
2. **什麼時候該收手**：牆在消退、或價格逼近
3. **要不要滾倉**：平掉現在的，改賣下一個合約更好的位置

## WHY

用戶觀察約一年：主動交易者在結算時幾乎總是落在獲利區間內。他們用真金白銀畫出來的天花板（call 側大量賣出的位置）是有效的價格屏障。如果能量化這面牆的位置、厚度、演變，covered call 的選位就從主觀判斷變成有數據支撐的決策。

## 核心前提

- 損益曲線反映「市場上所有主動交易力量的淨損益」（系統 top 1 精神）
- 主動交易者包含散戶、法人、被迫避險的 MM，不區分身份
- 倒 V 形不是一開始建好的靜態部位，是動態交易一路調整的累積結果
- 牆的有效性取決於口數厚度——薄牆會被輕鬆打穿

## WHAT — 需要建的東西

### 1. 盤中定時快照（Phase 1）
- **日盤**：09:00, 09:30, 10:00, 10:30, 11:00, 11:30, 12:00, 12:30, 13:00, 13:30（共 10 張/合約）
- **夜盤**：15:30, 16:00, 16:30, ..., 23:30, 00:00（共 18 張/合約）
- **收盤**：13:45 照舊不動
- **範圍**：所有追蹤中的 full series（帶 N 的，目前 3 週選 + 1 月選 = 4 個）
- **夜盤只存 full series**（日盤系列凍在 13:45 值，不浪費空間）
- **has_data 保護**：沒有實際交易資料時不存空檔
- **存放位置**：`snapshots/intraday/` 目錄，跟收盤快照分開
- **檔名格式**：`{series}_{YYYY-MM-DD}_{HHMM}.json`，例如 `TXUN04_2026-04-08_0930.json`
- **內容**：跟收盤快照一樣（完整 columnar table + pnl + raw_calls/raw_puts + atm + implied_forward），加上 `futures_price`
- **存量**：~30KB × 28 張 × 4 合約 ≈ 3.4MB/天，~100MB/月

### 2. 分鐘價格線（Phase 1）
- **頻率**：每 1 分鐘
- **記錄**：timestamp, futures_price, implied_forward（active series）
- **時間**：08:45~13:45（日盤）+ 15:00~00:00（夜盤，電腦開著就記）
- **存放**：`monitor/price_log_{YYYY-MM-DD}.csv`
- **格式**：CSV，一行一分鐘
- **存量**：~70KB/天，~2MB/月
- **實作**：main.py `_periodic_broadcast` 每 60 秒追加一行，零額外計算

### 3. 分析工具（Phase 3）
- 從快照提取特徵：牆的位置、厚度、距離 ATM、premium、跨日/盤中變化
- FITX 分鐘線與牆的位置疊圖：價格什麼時候靠近牆、被彈回還是穿過
- 跨合約牆比較：同價位不同合約的牆厚度 + premium 對比
- 當週累積曲線漸進變化（從 daily 快照的 raw_calls/raw_puts 事後重算）
- 統計分析：牆厚度 vs 結算時是否被打穿的勝率

### 4. Covered Call 決策規則（Phase 4）

**進場（賣哪裡）：**
- 找 call 側 net_call 負值最大且 vol 夠厚的履約價
- 同時看該位置的 premium（avg_price_call 或 bid_price_call），牆厚但 premium 太低不值得
- 最佳賣點 = 牆夠厚 × premium 值得收
- 距離 ATM 至少 X 點（X 待數據定義）
- 比較多個合約：如果下一個合約同位置牆更厚、premium 更好 → 賣下一個

**持有（安全嗎）：**
- 每天看牆的口數趨勢（穩定/變厚 = 安全，連續縮減 = 警告）
- FITX 分鐘線有沒有觸碰過牆的位置（用 high/low 判斷）
- 夜盤後牆是否還在（09:00 快照 vs 前一天收盤快照比對）
- 美股開盤後（21:30+）的牆變化特別注意

**平倉/滾倉（什麼時候走）：**
- 牆的口數降到初始的 X% 以下 → 平倉
- FITX 突破牆的位置 → 停損
- 當前合約剩餘時間價值很低 + 下一個合約有更好的牆和 premium → 滾倉
- 臨近結算、牆完整 → 不動，讓它到期

## HOW — 實作方式

### Phase 1：資料基建（改 code）

**main.py 改動：**
1. 新增 `_try_save_intraday_snapshot()`：定時存盤中快照到 `snapshots/intraday/`
   - 對所有 full series 存，不限 active
   - 包含 `futures_price` 欄位
   - 使用 columnar + compact JSON
   - has_data 保護
2. 修改 `_periodic_broadcast()`：
   - 每 1800 秒（30 分鐘）觸發盤中快照
   - 每 60 秒追加一行到分鐘價格 CSV
   - 判斷當前是否在交易時段（日盤 08:45~13:45 / 夜盤 15:00~00:00）

**不動的：**
- 13:45 收盤快照邏輯不變
- weekly_sum 只在結算日存（v4.15）
- xqfap_feed.py 不需要改
- 前端不需要改

### Phase 2：累積數據（4~6 週不動 code）
- 系統自動運行，每天收集盤中快照 + 分鐘價格線
- 目標：涵蓋 6~8 個週選結算 + 1~2 個月選結算
- 預估累積：~100MB 盤中快照 + ~2MB 分鐘線 + 原有收盤快照

### Phase 3：分析工具（Python 腳本）
- 讀取所有快照 + 分鐘線
- 提取牆特徵、價格路徑、結算結果
- 跑統計、找規律
- 輸出視覺化報告

### Phase 4：策略規則化
- 把 Phase 3 找到的規律寫成量化規則
- 在 OptionChart 前端加輔助標記（建議賣點、牆強度指標等）
- Paper trade 驗證

## 已知限制

| 限制 | 影響 | 應對 |
|------|------|------|
| 00:00~08:45 無數據（電腦關機） | 漏掉深夜低流動性時段 | 09:00 快照抓隔夜淨變化 |
| 無未平倉量（OI） | 只有每天的方向性流量，不是累積 OI | weekly 累積間接看多日累積；raw_calls/raw_puts 可事後重算 |
| 樣本量 | 目前只有 2 週數據（04W1 + 04F1） | Phase 2 累積 4~6 週再下結論 |
| implied_forward 非官方結算價 | 可能有微小偏差 | 用戶確認偏差可接受 |

## 待驗證假說（Phase 3 要回答的問題）

1. 牆的口數達到多少才算「有效」？門檻在哪？
2. 有效牆最終被打穿的比例是多少？
3. 價格靠近牆時，牆通常變厚（防守）還是變薄（撤退）？
4. 牆通常什麼時候形成？（開盤就有 vs 尾盤才建 vs 隔夜？）
5. 夜盤美股開盤後，牆的存活率？
6. 曲線不對稱度跟最終結算方向有沒有相關？
7. breakeven 被突破後，價格繼續走的幅度和假突破比例？
8. 最佳 covered call 賣點的 premium / 牆厚度比是多少？
9. 滾倉 vs 持有到期，哪種情況下滾倉更有利？
